#### 🔍 GD-72099 详细技术分析

**问题表：** `dcp_teacher_task_count` (ODPS 表)

**关键 SQL 逻辑：**
```sql
-- 在 dcp_teacher_task_count 节点的 INSERT OVERWRITE 语句中
SELECT 
    ...
    DATE(wt.start_date) count_date,  -- ⚠️ 关键：按任务开始时间统计
    ...
FROM dcp_workcell_tasks wt
...
GROUP BY DATE(wt.start_date), s.instid, s.media_owner_id, s.label_id
```

**根本原因：**
- SQL 中使用 `DATE(wt.start_date)` 作为 `count_date`
- 统计归属月份 = 任务开始日期所在月份
- 而非学员实际完成任务的日期

**示例：**
```
任务周期：2026-01-30 ~ 2026-03-01
学员完成时间：2026-02-15
统计归属：2026-01（因为 wt.start_date = 2026-01-30）
```

**相关表：**
- `dcp_teacher_task_count` - 老师任务统计表
- `dcp_workcell_tasks` - 任务表（含 start_date）
- `dcp_share` - 分享表
- `dcp_workcell_share_settings` - 任务设置表

## 🎯 如何回答类似问题

当用户问"XX 数据显示为 0"或"数据统计不对"时：

### 第一步：快速匹配 NAB 规则
```bash
python3 nab-classifier.py "数据为 0 有任务但没显示"
```

### 第二步：如果匹配到 GD-72099 类似场景

**回复模板：**
```
【Not A Bug - 数据统计口径】(参考：GD-72099)

问题原因：统计是按【任务开始时间】归属月份的，不是按完成时间。

技术原因：
- dcp_teacher_task_count 表使用 DATE(wt.start_date) 作为统计日期
- 任务开始于 A 月，学员 B 月完成 → 统计到 A 月

相关 Bug 单：GD-72099
- 机构：hwjxx
- 老师账号：15075496858
- 任务周期：1.30 ~ 3.1
- 统计归属：1 月（非 2 月）

建议：
1. 确认任务开始时间
2. 查看对应月份的统计数据
3. 如需调整统计口径，需产品评估修改 SQL 逻辑
```

### 第三步：如果不匹配
→ 收集更多信息（任务 ID、截图、时间）→ 人工排查 → 确认是新 NAB？→ 存档更新规则

1. **先分类，再排查** - 遇到问题先用工具分类
2. **匹配到 NAB 直接回复** - 不用每次都深入排查

4. **案例存档** - 确认过的 NAB 保存到 `nab-cases/`
